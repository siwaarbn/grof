## 1. Background & Context

### How GPU Execution Works
When a Python script runs `torch.matmul(a, b)`, PyTorch calls a C++ library (LibTorch), which eventually calls the CUDA Runtime API (e.g., `cudaLaunchKernel`). The CPU sends a command to the GPU command queue (stream) and continues execution (asynchronous).

### What is CUPTI?
The **CUDA Profiling Tools Interface (CUPTI)** is the official NVIDIA API for creating profiling tools.
- **Callback API:** Good for knowing *when* a function is called on CPU.
- **Activity API:** The "real" profiler. It buffers data on the GPU/Driver and gives you a batch of records later. It provides exact GPU start/end timestamps.

For Milestone 1, we will skip the basic "Callback" approach and go straight for the **Activity API**, as it's required for the final product.

### LD_PRELOAD
We want to profile PyTorch without recompiling PyTorch. We will build a shared library (`libgrof_cuda.so`) and force the Linux loader to load it before anything else using `LD_PRELOAD=./libgrof_cuda.so python script.py`.

## 2. Detailed Steps

### Week 1: Environment & Injection
1.  **Setup:** Verify CUDA Toolkit and build CUPTI samples.
2.  **Injection Harness:** Create a C++ project that compiles to a `.so`. Use `__attribute__((constructor))` to run your init code.
3.  **LD_PRELOAD Test:** Verify you can print "Hello from GROF" before PyTorch starts.

### Week 2: The Activity API
1.  **Initialization:** Initialize CUPTI and enable the `CUPTI_ACTIVITY_KIND_KERNEL` and `CUPTI_ACTIVITY_KIND_MEMCPY` kinds.
2.  **Buffering:** Implement the buffer request and completion callbacks. CUPTI needs you to allocate memory where it will store the trace records.
3.  **Processing:** Write the code to parse the returned buffers. You will get structs like `CUpti_ActivityKernel4`.

### Week 3: Data Extraction
1.  **Kernel Names:** Read the `name` field from the kernel record. You must demangle it (convert `_Z5k...` to `kernel_func`).
2.  **Timestamps:** Capture `start` and `end` timestamps. Note: These are usually in nanoseconds.
3.  **Streams:** Capture the `streamId`. This is vital for visualizing concurrency.
4.  **Memcpy:** Capture memory transfers (Host-to-Device, Device-to-Host). This helps identify bottlenecks where the GPU is waiting for data.

### Week 4: Correlation & Output
1.  **Correlation ID:** We need to link the CPU launch to the GPU execution. Use `cuptiActivityPushExternalCorrelationId` (Activity API) or `cuptiSubscribe` (Callback API) to push a unique ID when the kernel is launched on CPU.
2.  **JSON Output:** Instead of printing to stdout, write the collected records to a JSON file (`gpu_trace.json`) at the end of execution.
3.  **Format:** `[{ "name": "sgemm", "start": 100, "end": 200, "stream": 1 }, ...]`

---

## 3. Evaluation Metrics

To pass Milestone 1, your deliverable must meet these criteria:

1.  **Activity API:** You must use the Activity API, not just Callbacks.
2.  **Completeness:** The trace must include **Kernels** AND **Memory Copies**.
3.  **Timing:** You must report the **duration** of each kernel (End - Start).
4.  **Concurrency:** If two kernels run on different streams, your data must reflect that (different stream IDs).
5.  **No Crashes:** PyTorch must run to completion.
