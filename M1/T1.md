## 1. Background & Context

### What is eBPF?
Extended Berkeley Packet Filter (eBPF) is a technology that allows you to run sandboxed programs in the Linux kernel without changing kernel source code or loading kernel modules. It is event-driven; we can attach eBPF programs to kprobes (kernel functions), uprobes (user-space functions), or tracepoints.

### Why eBPF for Python Profiling?
Standard Python profilers (like `cProfile`) use the `sys.settrace` hook, which adds significant overhead because it interrupts the interpreter for every function call.
To achieve **<5% overhead**, we need a "sampling profiler". Instead of stopping every call, we will wake up 100 times per second (100Hz), look at what the Python interpreter is doing, record it, and go back to sleep.

### CPython Internals
Python is an interpreted language. The CPython interpreter maintains a linked list of "PyFrameObjects" representing the call stack.
- The main execution loop is often in a C function called `PyEval_EvalFrameDefault` (or similar, depending on Python version).
- We need to read the `PyThreadState`, find the current `PyFrameObject`, and traverse the pointers to get the function name and line number.

## 2. Detailed Steps

### Week 1: Environment & "Hello World"
1.  **Setup Linux Environment:** Ensure you have a Linux machine (VM or bare metal) with root access. eBPF requires `sudo`.
2.  **Install BCC (BPF Compiler Collection):** This is the toolkit we will use to write eBPF programs in C and load them via Python.
3.  **Basic Uprobe:** Write a BCC script that attaches a `uprobe` to `PyEval_EvalFrameDefault`. Print a message every time a Python function is called.
4.  **Symbol Resolution:** Use `nm` or `objdump` to verify you can find the symbol addresses in the `python` binary.

### Week 2: Deep Stack Walking
1.  **Struct Definition:** Define the CPython structs (`PyFrameObject`, `PyCodeObject`, `PyUnicodeObject`) inside your BPF C program. You'll need to handle offsets manually or include the correct headers.
2.  **Pointer Chasing:** Implement the logic to read `frame->f_back`.
3.  **Looping:** Write a BPF loop (unrolled, as BPF loops are limited) to walk up the stack up to 20-30 frames deep.
4.  **Output:** Print the full stack trace (e.g., `main;train;forward;conv2d`) to the trace pipe.

### Week 3: Kernel-Side Aggregation (Performance)
*Crucial for low overhead.*
1.  **BPF Maps:** Instead of sending every sample to user-space (slow), use a `BPF_HASH` map.
2.  **Key-Value:**
    - **Key:** The stack trace ID (or the full stack trace hash).
    - **Value:** A counter (how many times we saw this stack).
3.  **Logic:** When the timer fires:
    - Capture stack.
    - Hash it.
    - Increment counter in Map.
4.  **User-Space:** The Python wrapper only reads the Map once every few seconds (or at the end), drastically reducing overhead.

### Week 4: Handling Threads & Native Code
1.  **Multi-threading:** Ensure your profiler works when the Python script uses `threading.Thread`. You need to filter by PID/TID.
2.  **Native Frames:** (Stretch Goal) Can you also capture the C++ stack frames (e.g., inside PyTorch C++ code) using `bpf_get_stackid`? Try to mix Python frames and Kernel frames.
3.  **Integration:** Output the data in a format that T4 (Backend) can accept (JSON).

## 3. Evaluation Metrics

To pass Milestone 1, your deliverable must meet these criteria:

1.  **Depth:** The profiler must capture the **full call stack**, not just the top function.
2.  **Performance:** It must use **BPF Maps** for aggregation. Streaming events to user-space at 100Hz is not acceptable.
3.  **Accuracy:** Compare your output against `py-spy` (an existing tool). The flamegraphs should look similar.
4.  **Robustness:** It must handle a recursive function (deep stack) without crashing the BPF verifier.
