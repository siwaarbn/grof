# Milestone 2: Interactive Correlation Views & Performance Insights (T5)

**Team:** Siwar, Olga
**Duration:** 4 Weeks
**Goal:** Build a unified CPU-GPU correlation view, implement a critical path visualization, add comparison mode for multiple runs, and create an interactive performance insights dashboard.

---

## 1. Background & Context

### From Separate to Unified
In Milestone 1, we built:
- **Flamegraph:** Shows CPU-side execution.
- **Timeline:** Shows GPU-side execution.

But they were **disconnected**. Users had to mentally correlate them.

In Milestone 2, we'll create **integrated views** that show the full story:
- "When I click a Python function in the flamegraph, highlight the GPU kernels it launched."
- "When I hover over a GPU kernel in the timeline, show the CPU stack that triggered it."

### Performance Insights
Beyond raw visualization, users need **actionable insights**:
- "Your model is CPU-bound. The GPU is idle 40% of the time."
- "Memory transfers are the bottleneck. Consider using pinned memory."
- "Kernel X has low SM utilization. Try increasing batch size."

This requires **automated analysis** of the trace data.

---

## 2. Detailed Steps

### Week 1: Unified CPU-GPU Correlation View
1.  **Linked Interactions:** When clicking a node in the flamegraph:
    - Highlight the corresponding region in the GPU timeline.
    - Show a tooltip: "This function launched 42 kernels consuming 150ms GPU time."
2.  **Reverse Correlation:** When clicking a GPU kernel in the timeline:
    - Expand the flamegraph to show the CPU stack that launched it.
    - Draw a visual "thread" connecting the two views.
3.  **Implementation:** Use React Context or a state management library (Zustand/Redux) to sync the two visualizations.

### Week 2: Critical Path Visualization
*The most important feature for optimization.*
1.  **What is the Critical Path?** The longest chain of dependencies from start to finish. This determines the minimum possible runtime.
2.  **Data:** Fetch from T4's `/sessions/{id}/critical-path` API.
3.  **Visualization:** Draw a "dependency graph" or annotate the timeline with the critical path in a distinct color (e.g., red).
4.  **Insight:** Display: "The critical path is 850ms. Your total runtime is 900ms. There's only 50ms of potential parallelism improvement."

### Week 3: Comparison Mode (A/B Testing for Performance)
*Essential for optimization workflows.*
1.  **Multi-Session Selection:** Allow users to select 2+ profiling sessions from the dashboard.
2.  **Side-by-Side View:** Display flamegraphs or timelines side-by-side.
3.  **Diff Highlighting:** Highlight functions that got slower or faster.
    - Use color coding: Green = faster, Red = slower.
4.  **Metrics Table:** Show a table comparing key metrics:
    | Metric | Run A | Run B | Delta |
    |--------|-------|-------|-------|
    | Total Time | 1000ms | 950ms | -5% ✓ |
    | GPU Time | 800ms | 820ms | +2.5% ✗ |

### Week 4: Performance Insights Dashboard
*Automated advice for users.*
1.  **Bottleneck Detection:** Implement automatic analysis:
    - If `(GPU idle time / total time) > 30%`: Show warning "CPU Bottleneck Detected".
    - If `(memory transfer time / total time) > 20%`: Suggest "Optimize data transfers".
2.  **Kernel Analysis:** For each kernel, show:
    - SM Utilization (from T2's metrics).
    - If low: Suggest "Increase batch size or grid dimensions".
3.  **Recommendations Panel:** Add a collapsible panel on the side with a ranked list of optimization opportunities.
4.  **Export Report:** Allow users to export a PDF performance report with graphs and recommendations.

## 3. Evaluation Metrics

To pass Milestone 2, your deliverable must meet these criteria:

1.  **Bidirectional Correlation:** Clicking in the flamegraph must highlight the GPU timeline, and vice versa.
2.  **Critical Path:** The critical path must be visually distinguishable in the timeline view.
3.  **Comparison Mode:** Users must be able to load 2 sessions and see a visual diff.
4.  **Insights:** The dashboard must display at least 3 types of automated insights (CPU bottleneck, memory bottleneck, low SM utilization).
5.  **Performance:** The UI must remain responsive (60 fps) even with 100,000+ events loaded.

If your team completes the core deliverables ahead of schedule, consider these advanced features:
