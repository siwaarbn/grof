**Goal:** Integrate CUPTI's native correlation mechanism, add detailed GPU metrics (SM utilization, memory bandwidth), and implement stream synchronization tracking.

## 1. Background & Context

### The Correlation Challenge
In Milestone 1, we captured GPU kernel execution, but we couldn't link it back to the CPU code that launched it. CUPTI provides two mechanisms for correlation:

1.  **External Correlation IDs:** We can push custom IDs using `cuptiActivityPushExternalCorrelationId()`. This must match the IDs that T1 (eBPF) generates.
2.  **Native Correlation:** CUPTI automatically generates correlation IDs between CPU API calls and GPU activities. We can use this as a fallback.

### Beyond Basic Timing
For production profiling, users need more than "kernel X took 5ms". They need:
- **SM Utilization:** What percentage of streaming multiprocessors were active?
- **Memory Bandwidth:** Was the kernel compute-bound or memory-bound?
- **Occupancy:** How many warps were active per SM?
- **Synchronization:** When did `cudaStreamSynchronize()` block the CPU?

CUPTI provides these via the **Metrics API** and **PC Sampling API** (advanced).

## 2. Detailed Steps

### Week 1: External Correlation Integration
1.  **Callback Subscription:** Use `cuptiSubscribe()` to register callbacks for `CUPTI_RUNTIME_TRACE_CBID_cudaLaunchKernel_v7000`.
2.  **Receive Correlation ID from T1:** T1 will set a correlation ID when a CUDA API is called. Options:
    - **Shared Memory:** Use a shared memory region (e.g., `/dev/shm/grof_correlation`).
    - **Environment Variable:** Pass it via thread-local storage (TLS).
3.  **Push to CUPTI:** Call `cuptiActivityPushExternalCorrelationId(correlation_id)` in the callback.
4.  **Verification:** Ensure the GPU activity records now have the correct `externalId` field.

### Week 2: Advanced Metrics Collection
1.  **Enable Metric Collection:** Use `cuptiMetricGetIdFromName()` to get metric IDs for:
    - `achieved_occupancy`
    - `sm_efficiency`
    - `dram_utilization`
2.  **Kernel-Specific Metrics:** Use the **Profiling API** (`cuptiProfilerSetConfig`) to collect these metrics for specific kernels.
3.  **Store Metrics:** Extend the JSON output to include:
    ```json
    {
      "name": "sgemm",
      "duration_ns": 5000000,
      "sm_efficiency": 85.3,
      "dram_utilization": 42.1
    }
    ```

### Week 3: Stream Synchronization Tracking
*Critical for understanding CPU-GPU dependencies.*
1.  **Track Synchronization Calls:** Intercept `cudaStreamSynchronize` and `cudaDeviceSynchronize`.
2.  **Measure Wait Time:** Record the timestamp when the synchronize call starts and ends (the time the CPU was blocked).
3.  **Event Correlation:** Link the synchronize event to the stream/kernel it was waiting for.
4.  **Output:** Add a new event type:
    ```json
    {
      "type": "synchronize",
      "stream_id": 1,
      "start": 100,
      "end": 150,
      "waited_for": "kernel_name"
    }
    ```

### Week 4: Memory Transfer Analysis
1.  **Detailed Memcpy:** Extend the Activity API to capture more details about memory transfers:
    - Source (Host/Device), Destination, Size, Bandwidth.
2.  **Pageable vs Pinned:** Detect if the transfer used pinned memory (much faster).
3.  **Overlap Analysis:** Identify when kernel execution and memory transfers overlap (good for performance).
4.  **Visualization Data:** Provide data for T5 to show memory transfers on the timeline.

## 3. Evaluation Metrics

To pass Milestone 2, your deliverable must meet these criteria:

1.  **Correlation IDs:** GPU events must have the **same correlation ID** as the CPU events from T1.
2.  **Metrics:** You must report at least **2 advanced metrics** (SM efficiency and DRAM utilization).
3.  **Synchronization:** You must capture `cudaStreamSynchronize()` events with their duration.
4.  **Memory Analysis:** Memory transfer events must include transfer size and bandwidth.
5.  **Format Compliance:** The output must match the schema expected by T4 (Backend).

## 4. Stretch Goals (If Finished Early)

If your team completes the core deliverables ahead of schedule, consider these advanced features:

1.  **PC Sampling (Instruction-Level Profiling):** Use CUPTI's PC Sampling API to capture which SASS instructions are executed most frequently.
2.  **Tensor Core Usage Detection:** Identify which kernels use Tensor Cores (critical for A100/H100 GPUs).
3.  **NVLink Bandwidth Tracking:** Capture inter-GPU communication bandwidth for multi-GPU setups.
4.  **Kernel Replay:** Save kernel parameters to enable "replay" of specific kernels for debugging.
5.  **Power Consumption:** Integrate NVML to track GPU power draw during execution (correlate high power with specific kernels).
6.  **CUDA Graph Detection:** Identify when CUDA Graphs are being used and measure their overhead savings.
