**Team:** Fouad, Maria
**Duration:** 4 Weeks
**Goal:** Extend the eBPF profiler to emit Correlation IDs when CUDA API calls are made, and integrate native C/C++ stack traces alongside Python frames.

---

## 1. Background & Context

### The Missing Link
In Milestone 1, we built two independent profilers:
- **T1 (eBPF):** Captures Python stack traces.
- **T2 (CUDA):** Captures GPU kernel execution.

But we don't know **which Python function triggered which GPU kernel**. This is the **Correlation Problem**.

### The Solution: Correlation IDs
When a Python function calls `torch.matmul()`, which eventually calls `cudaLaunchKernel()`, we need to:
1.  **CPU Side (eBPF):** Attach a `uprobe` to CUDA API entry points (e.g., `cudaLaunchKernel`).
2.  **Generate a Unique ID:** Create a correlation ID (e.g., incrementing counter or timestamp).
3.  **Emit Both:** Send both the **Python stack trace** AND the **Correlation ID** to user-space.
4.  **GPU Side (CUPTI):** T2's CUPTI code must emit the **same Correlation ID** when it records GPU kernel execution.

This allows T4 (Backend) to join the CPU and GPU traces.

### Native Stack Traces
PyTorch's heavy lifting happens in C++ (LibTorch). To get the full picture, we need to capture **both**:
- Python frames (`train() → forward() → matmul()`)
- Native frames (`at::matmul → cudaLaunchKernel`)

We'll use the BPF helper `bpf_get_stackid()` to capture native frames and merge them with the Python frames.

## 2. Detailed Steps

### Week 1: CUDA API Interception with uprobes
1.  **Symbol Discovery:** Use `nm` or `objdump` to find CUDA API symbols in `libcudart.so`.
    - Target: `cudaLaunchKernel`, `cudaMemcpyAsync`, `cudaStreamSynchronize`.
2.  **Attach Uprobes:** Extend your BCC script to attach uprobes to these functions.
3.  **Correlation ID Generation:** When a CUDA API is called:
    - Read the current timestamp or use an atomic counter.
    - Store it in a BPF Map: `cuda_correlation[tid] = correlation_id`.

### Week 2: Emit Correlation Events
1.  **New Event Type:** Define a new event struct:
    ```c
    struct correlation_event {
        u64 timestamp;
        u32 pid;
        u32 tid;
        u64 correlation_id;
        char stack_trace[256];  // Simplified stack
    };
    ```
2.  **Perf Buffer:** Send these events to user-space via `BPF_PERF_OUTPUT`.
3.  **User-Space Handler:** Modify your Python wrapper to consume these events and write them to a file (`cpu_correlation.json`).

### Week 3: Native Stack Integration
1.  **BPF Stack Traces:** Use `bpf_get_stackid()` to capture the native call stack.
2.  **Symbol Resolution:** In user-space, resolve the addresses to function names using `/proc/pid/maps` and `addr2line` or a symbol library.
3.  **Merge Logic:** Combine the Python frames (from `PyFrameObject` walking) with the native frames:
    - Example Output: `[Python] train → forward → [Native] at::matmul → cudaLaunchKernel`.

### Week 4: Performance Tuning & Testing
1.  **Overhead Measurement:** Work with T3 to measure the overhead of the new correlation logic. Target: Still <5%.
2.  **Multi-threaded Testing:** Ensure correlation IDs are thread-local (use `bpf_get_current_pid_tgid()` correctly).
3.  **Integration:** Ensure the output format matches what T4 expects for correlation matching.
4.  **Documentation:** Write detailed docs on how the correlation ID flows through the system.

## 3. Evaluation Metrics

To pass Milestone 2, your deliverable must meet these criteria:

1.  **Correlation Events:** You must emit CPU-side correlation events when CUDA APIs are called.
2.  **Native Frames:** The stack traces must include both Python AND C++ frames.
3.  **Thread Safety:** Correlation IDs must be unique per thread.
4.  **Performance:** The total overhead must still be <5% (validated by T3).
5.  **Output Format:** The JSON output must include:
    ```json
    {
      "type": "correlation",
      "timestamp": 123456789,
      "tid": 1234,
      "correlation_id": 42,
      "stack": ["train", "forward", "at::matmul", "cudaLaunchKernel"]
    }
    ```
