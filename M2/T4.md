# Milestone 2: CPU-GPU Correlation Engine & Query Optimization (T4)

**Team:** Khalil, Siwar
**Duration:** 4 Weeks
**Goal:** Build the correlation engine that joins CPU and GPU traces, implement advanced query APIs for trace analysis, and optimize database performance for million-event workloads.

---

## 1. Background & Context

### The Correlation Problem
After Milestone 1, we have:
- **CPU Traces:** Python/C++ stack traces from eBPF (T1).
- **GPU Traces:** Kernel execution timelines from CUPTI (T2).

But they're in separate files. The Backend must **join** them using the **Correlation IDs** generated by T1 and T2.

### The Challenge
- **Volume:** A 10-second run can generate 10,000+ CPU samples and 100,000+ GPU events.
- **Timing:** CPU and GPU timestamps come from different clocks. We need clock synchronization.
- **Indexing:** Querying "Show me all GPU kernels launched from function X" must be fast (<1 second).

## 2. Detailed Steps

### Week 1: Correlation Data Model & Ingestion
1.  **Extended Schema:** Add correlation tables:
    - `CorrelationEvent`: (id, session_id, correlation_id, cpu_timestamp, cpu_stack_hash, gpu_kernel_id).
    - Update `GpuEvent` to include `correlation_id` (foreign key).
2.  **Dual Ingestion:** Implement endpoints:
    - `POST /sessions/{id}/cpu-correlation` (from T1).
    - `POST /sessions/{id}/gpu-events` (from T2, now with correlation IDs).
3.  **Batch Processing:** Use bulk inserts (e.g., `INSERT INTO ... VALUES (...)` with multiple rows) for performance.

### Week 2: The Correlation Engine
1.  **Matching Logic:** Write a background job that runs after ingestion:
    - Query all `CorrelationEvent` records for a session.
    - For each GPU event with a `correlation_id`, find the matching CPU stack.
    - Create a join table or materialized view linking them.
2.  **Clock Synchronization:** CPU timestamps (eBPF) and GPU timestamps (CUPTI) use different clocks.
    - **Solution:** Use `cudaEventCreate` + `cudaEventRecord` at profiling start to create a "sync point" between CPU and GPU time.
    - Store the offset: `gpu_time = cpu_time + offset`.
3.  **Validation:** Test on a simple PyTorch script. Verify that `torch.matmul` in Python is correctly linked to the `sgemm` kernel on GPU.

### Week 3: Advanced Query APIs
*The Frontend needs more than raw data.*
1.  **Critical Path Analysis:** Implement `GET /sessions/{id}/critical-path`.
    - **Logic:** Find the sequence of CPU and GPU operations that determines the total runtime (the longest dependency chain).
    - This is a graph problem (DAG traversal).
2.  **Function-Level Breakdown:** Implement `GET /sessions/{id}/functions`.
    - **Query:** Group by function name, sum total GPU time triggered by each function.
    - **Output:** `{ "train": { "total_gpu_time_ms": 500, "kernel_count": 1200 }, ... }`
3.  **Timeline Range Query:** Implement `GET /sessions/{id}/timeline?start=100&end=200`.
    - Return only events in a specific time range (for zooming in the UI).

### Week 4: Database Optimization & Caching
1.  **Indexing:** Add indexes on:
    - `correlation_id` (for joins).
    - `timestamp` (for range queries).
    - `session_id` + `function_name` (for grouping).
2.  **Query Optimization:** Use `EXPLAIN ANALYZE` to optimize slow queries.
3.  **Caching:** Implement Redis caching for expensive queries (e.g., Flamegraph generation).
    - Cache key: `flamegraph:{session_id}`.
    - Invalidate cache if new data is ingested.
4.  **Load Testing:** Use `locust` or `ab` (ApacheBench) to simulate 100 concurrent requests. Ensure response time <500ms.

## 3. Evaluation Metrics

To pass Milestone 2, your deliverable must meet these criteria:

1.  **Correlation Success:** At least 90% of GPU kernels must be successfully matched to a CPU stack trace.
2.  **Query Performance:** The critical path API must return results in <2 seconds for a 10-second trace.
3.  **Scalability:** The system must handle a trace with 1 million events without crashing.
4.  **API Completeness:** You must implement at least 3 advanced query endpoints (Critical Path, Function Breakdown, Timeline Range).
5.  **Documentation:** API docs (OpenAPI/Swagger) must be auto-generated and accessible at `/docs`.

## 4. Stretch Goals (If Finished Early)

If your team completes the core deliverables ahead of schedule, consider these advanced features:

1.  **Real-Time Streaming:** Implement WebSocket endpoints that stream live profiling data to the UI during execution.
2.  **Automatic Anomaly Detection:** Use statistical methods to flag "abnormal" traces (e.g., unusually long kernel, unexpected synchronization).
3.  **Query Language:** Implement a simple query DSL (Domain Specific Language) for trace analysis: `SELECT kernels WHERE duration > 5ms AND sm_efficiency < 50%`.
4.  **Delta Storage:** Store only differences between consecutive runs (compression technique) to save database space.
5.  **Distributed Tracing:** Extend to support multi-node distributed training with trace correlation across machines.
6.  **Graph Database Integration:** Use Neo4j for storing the call graph, enabling complex graph queries like "Find all paths from function X to kernel Y".
