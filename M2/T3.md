## 1. Background & Context

### The 5% Promise
The core value proposition of GROF is "low overhead profiling". But claiming <5% overhead requires proof.

### What Counts as Overhead?
- **Runtime Overhead:** Does the code run slower when profiled?
- **Memory Overhead:** Does the profiler use excessive RAM?
- **Data Collection Overhead:** How much CPU time is spent processing the trace data?

### The Comparison Baseline
We need to compare GROF against:
1.  **No Profiling:** The pure baseline.
2.  **Nsys:** NVIDIA's official profiler (typically 10-30% overhead).
3.  **Py-spy:** A popular Python sampling profiler (~1-2% overhead, but no GPU).

This will position GROF's performance in the landscape.

## 2. Detailed Steps

### Week 1: Automated Overhead Measurement Framework
1.  **Three-Way Runner:** Extend `run_suite.py` from M1 to support three modes:
    - `--mode=baseline` (No profiling)
    - `--mode=grof` (With GROF enabled)
    - `--mode=nsys` (With `nsys profile`)
2.  **Controlled Environment:** Disable CPU frequency scaling (`sudo cpupower frequency-set -g performance`) to reduce variance.
3.  **Warmup:** Always run 3 warmup iterations before measuring.
4.  **Sample Size:** Run each configuration N=30 times (sufficient for statistical significance).

### Week 2: Deep Overhead Analysis
1.  **Per-Component Breakdown:** Measure the overhead of individual components:
    - **eBPF Only:** Run with just T1's profiler (no CUDA).
    - **CUPTI Only:** Run with just T2's profiler (no eBPF).
    - **Full System:** Both components enabled.
2.  **Attribution:** This tells us "where" the overhead comes from. Is it the eBPF sampling? The CUPTI buffering?
3.  **Memory Profiling:** Use `/usr/bin/time -v` or `psrecord` to track peak memory usage.

### Week 3: Comparative Study (GROF vs Industry Tools)
1.  **Nsys Comparison:** Profile the same workloads with Nsys.
    - Compare: Runtime, Trace File Size, Level of Detail.
2.  **Py-spy Comparison:** Run py-spy on the CPU-only benchmarks.
    - Question: Is GROF's eBPF profiler competitive with py-spy on CPU-only code?
3.  **Report:** Create a detailed comparison table:
    | Tool | Runtime Overhead | Memory Overhead | GPU Support | CPU Stack Depth |
    |------|------------------|-----------------|-------------|-----------------|
    | GROF | X% | YMB | ✓ | 30 |
    | Nsys | X% | YMB | ✓ | Limited |
    | Py-spy | X% | YMB | ✗ | 30 |

### Week 4: Continuous Integration & Regression Tests
1.  **GitHub Actions / GitLab CI:** Set up CI pipeline that runs the overhead benchmarks automatically on every commit.
2.  **Performance Regression Detection:** If a new commit increases overhead by >2%, fail the CI build.
3.  **Historical Tracking:** Store benchmark results in a database or CSV to track overhead trends over time.
4.  **Dashboard:** (Optional) Create a simple webpage that plots overhead over commits.

## 3. Evaluation Metrics

To pass Milestone 2, your deliverable must meet these criteria:

1.  **Overhead Report:** A detailed report showing GROF's overhead is <5% on at least 3 macro-benchmarks (ResNet, BERT, etc.).
2.  **Statistical Rigor:** All numbers must include 95% confidence intervals.
3.  **Comparison:** A side-by-side comparison with Nsys showing GROF is competitive or better.
4.  **Regression Tests:** CI must run automatically and fail if overhead exceeds 7% (allowing 2% buffer).
5.  **Documentation:** A "Performance Characteristics" document that users can reference.
